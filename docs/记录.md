## 问题 1：MatMul Kernel 架构分析

### 面试官提问
看你的 `matmul_kernel.cu`，你采用了 **1个block处理1行输出** 的策略（`ROW_PER_BLOCK=1`），每个block 128个线程，使用 `float4` 向量化加载和 CUB BlockReduce 完成归约。

**问题：请分析这个设计在 LLM 推理场景下的性能瓶颈，并说明为什么这不是一个高效的设计？**

1. **Occupancy**：这种设计对 GPU 占用率的影响？
2. **Memory Access Pattern**：weight 矩阵的访问模式有什么问题？
3. **与 cuBLAS / CUTLASS 的差距**：相比成熟的 GEMM 实现，你的方案缺少了哪些关键优化？

### 候选人回答
> 感谢面试官的提问，我承认这个矩阵乘法上有很多不足，首先是计算强度过低，导致算子的瓶颈都集中在访存上，这是 memory-bound 的，因为这是一个练手项目，所以没有做过多的优化，例如通过 swizzle 避免 bank conflict，以及更换精度使用 tensor core，还有常见的双缓存等都可以添加

### 评分：4/10

### 评价
回答过于笼统，没有针对三个维度逐一分析。Staff 级别需要更深入、更量化的分析能力。

| 维度 | 候选人回答 | 问题 |
|------|----------|------|
| Occupancy | ❌ 未回答 | 核心问题 |
| Memory Access Pattern | ⚠️ 只说 memory-bound | 没分析具体访问模式 |
| 与 cuBLAS/CUTLASS 差距 | ⚠️ 列举了技术名词 | 没解释为什么需要这些优化 |

**关键遗漏点**：
1. **Occupancy**：1 block = 1 row output，128 threads/block。假设 K=4096，只能启动 4096 个 block。RTX 3060 有 28 个 SM，每个 SM 最多 1536 threads。理论 occupancy 只有约 4096×128 / (28×1536) ≈ 12%，严重不足。

2. **Weight 访问模式**：每个 block 读取同一行 weight 的全部元素，但不同 block 读取不同行。对于列主序存储会导致严重的 **non-coalesced access**。

3. **缺少的关键优化**：没有分块到 shared memory。cuBLAS/CUTLASS 的核心是将 weight 和 input 分块加载到 shared memory，利用数据复用。

## 问题 2：MatMul Kernel 定量分析

### 面试官追问
假设 MatMul 用于 LLM 推理的 Linear 层，输入 shape 为 `(1, 4096)`，weight shape 为 `(4096, 4096)`。

用当前 kernel 配置 `<<<K, 128>>>`：
1. 计算实际启动的 block 数量？
2. 计算每个 thread 需要处理的元素数量？
3. 计算每个 thread 的 FMA 与 global memory 访问量之比（Arithmetic Intensity）？
4. 对比 RTX 3060 的 Roofline，kernel 理论上受限于算力还是带宽？能达到峰值性能的百分之几？

### 候选人回答
> 实际启动的是 4096 个 block，每个 thread 处理 4096/128 = 32 个元素，FMA=2*4*32，访问量为 32*4

### 评分：3/10

### 评价
计算有明显错误，且没有完成 Roofline 分析。

| 问题 | 候选人回答 | 正确答案 |
|------|----------|----------|
| Block 数量 | 4096 ✓ | 4096 ✓ |
| 每个 thread 处理元素 | 32 ✓ | 32 ✓ |
| FMA 数量 | 2×4×32=256 ❌ | **32 FMA** |
| 访存量 | 32×4=128 bytes ❌ | **256 bytes** |
| Arithmetic Intensity | ❌ 未计算 | **0.125 FLOPS/byte** |
| Roofline 分析 | ❌ 未回答 | 见下方 |

### 标准参考答案

**1. Block 数量**：`K = 4096` ✓

**2. 每个 thread 处理元素**：
```
pack_num = M / 4 = 4096 / 4 = 1024
每个 thread 处理 pack 数 = 1024 / 128 = 8 packs
每个 thread 处理元素数 = 8 × 4 = 32 ✓
```

**3. FMA 与访存量**：
```
每个 thread：
- 访问 input：32 × 4 bytes = 128 bytes
- 访问 weight：32 × 4 bytes = 128 bytes
- 总访存量：256 bytes

计算量：32 次乘法（归约可忽略）
Arithmetic Intensity = 32 FLOPS / 256 bytes = 0.125 FLOPS/byte
```

**4. Roofline 分析（RTX 3060）**：
```
峰值 FP32 算力：~10 TFLOPS
HBM 带宽：~360 GB/s
Ridge Point = 10 TFLOPS / 360 GB/s ≈ 27.8 FLOPS/byte

kernel AI：0.125 FLOPS/byte << 27.8 FLOPS/byte
→ 严重 Memory-Bound

理论上限性能 = 0.125 × 360 GB/s = 45 GFLOPS
峰值利用率 = 45 / 10000 = 0.45%
```

**结论：kernel 理论上只能达到峰值性能的 0.45%，差距 200 倍以上。**

## 问题 3：CUTLASS GEMM 分块策略

### 面试官追问
请描述 CUTLASS/cuBLAS 中 GEMM 的标准分块策略（Tiling Strategy）：

1. **Block 级分块**：一个 thread block 负责输出矩阵的多少元素？
2. **Warp 级分块**：一个 warp 负责多少元素？
3. **Shared Memory 使用**：A/B 矩阵如何分块加载？double buffering 如何工作？
4. **Register Tiling**：每个 thread 持有哪些数据在 register 中？
5. **为什么能提升 Arithmetic Intensity？**

### 候选人回答
> 一个 thread 负责一个 tile，典型的 tile 大小是 128*128，一个 warp 负责 16*16，设Bm Bn=128,Bk=8,那么 shared memory 就应该是 [Bm][Bk] , [Bn][Bk]，当然还可以转置，double buffering 是利用计算延迟覆盖读取延迟，在计算的时候可以读取下一批数据到 shared memory；通常使用矩阵外积，所以寄存器里可以存每一次外积的结果，逐步累加，最后就是最终写到目标矩阵中的值，按上面的假设，那么大小应该是[8][8]；这种设计可以显著降低从 global mem 访问的压力，同时利用寄存器，也减轻了访问 smem 的压力；再通过double buffering，进一步提高了计算强度

### 评分：5.5/10

### 评价
方向正确，但概念混淆，细节不够精确。

| 维度 | 候选人回答 | 问题 |
|------|----------|------|
| Block Tile | "一个 thread 负责 128×128" ❌ | **一个 thread block** 负责 128×128 |
| Warp Tile | 16×16 ⚠️ | 方向对，但没说清楚 warp 内线程分工 |
| Shared Memory | Bm×Bk, Bn×Bk ✓ | 正确 |
| Double Buffering | "计算延迟覆盖读取延迟" ⚠️ | 概念对，表述不精确 |
| Register Tiling | [8][8] ⚠️ | 数字对，但没解释清楚外积累加 |
| Arithmetic Intensity | "降低 global mem 压力" ⚠️ | 缺少定量分析 |

### 标准参考答案：CUTLASS GEMM 分块策略

```
Block 级分块（128×128 output）：
- 一个 thread block 负责 128×128 的输出 tile
- Block 数量 = (M/128) × (N/128)

Warp 级分块（16×16 output）：
- 一个 warp（32 threads）负责 16×16 或 32×8 的输出 tile
- 每个 block 有 64 个 warp（128×128 / 16×16 = 64）

Thread 级分块（8×8 output）：
- 每个 thread 计算 8×8 的输出 tile
- 寄存器：regA[8], regB[8], regC[8][8]

Shared Memory：
- sA[Bm][Bk] = sA[128][32]
- sB[Bk][Bn] = sB[32][128]
- Double buffering：2× 空间，计算 tile(k) 时加载 tile(k+1)

Arithmetic Intensity 计算（Bm=128, Bn=128, Bk=32）：
- 计算量：2 × 128 × 128 × 32 = 1,048,576 FLOPS
- 访存量：(128×32 + 32×128) × 4 = 32,768 bytes
- AI = 1,048,576 / 32,768 = 32 FLOPS/byte
- 接近 RTX 3060 ridge point（27.8）！
```

## 问题 4：LLM 推理 GEMM 特征

### 面试官追问
LLM 推理中的 MatMul 与传统训练场景有何不同？

1. **Prefill 阶段** vs **Decode 阶段**：输入 shape 的差异？哪个 compute-bound，哪个 memory-bound？
2. **Batch=1 Decode**：weight 矩阵访问模式特点？为什么推理是 bandwidth-bound？
3. **Int8 量化**：对 decode 阶段加速效果如何？还有哪些针对小 batch 的优化策略？

### 候选人回答
> 1. prefille 阶段输入的是所有 token，长度为 seq_len，也就是 ffn_dim，decode 结算输入的是一个 token；前者是 compute-bound,后者是 memory-bound 的。
> 2. batch=1 的时候 weight 矩阵会出现列访问的情况，推理就是 decode 阶段，是 bandwidth-bound 的
> 3. int8 量化加速一般，因为压力不在计算上；除了量化还可以做 pd 分离

### 评分：5/10

### 评价
核心概念理解正确，但术语混淆，缺少定量分析，优化策略回答偏离问题。

| 维度 | 候选人回答 | 问题 |
|------|----------|------|
| Prefill vs Decode | compute-bound / memory-bound ✓ | seq_len ≠ ffn_dim，概念混淆 |
| Batch=1 weight 访问 | "列访问" ⚠️ | 核心是 **weight 复用率极低** |
| Int8 量化效果 | "加速一般" ✓ | 正确，但没解释原因 |
| 其他优化 | "pd 分离" ⚠️ | 系统级优化，问的是 kernel 级优化 |

### 标准参考答案

**1. Prefill vs Decode 定量分析**

```
Prefill (seq_len = 512, batch = 1):
- 输入: [1, 512, 4096]
- 计算量: 2 × 512 × 4096 × 12288 ≈ 51.5 GFLOPS
- 访存量: 512×4096×4 + 4096×12288×4 ≈ 201 MB
- AI ≈ 256 FLOPS/byte → Compute-bound

Decode (batch = 1, seq_len = 1):
- 输入: [1, 1, 4096]
- 计算量: 2 × 1 × 4096 × 12288 ≈ 100 MFLOPS
- 访存量: 4096×4 + 4096×12288×4 ≈ 192 MB
- AI ≈ 0.5 FLOPS/byte → Memory-bound
```

**核心洞察**：Decode 阶段，每个 token 都要读取整个 weight 矩阵（192 MB），但只做 100 MFLOPS 计算。Weight 完全无法复用！

**2. Int8 量化效果有限的原因**：
- 计算加速: 2-4×（如果使用 Tensor Core INT8）
- 但 Decode 瓶颈是带宽
- 反量化开销可能抵消收益
- 实际加速效果：1.5-2×

**3. Kernel 级优化策略（针对小 Batch Decode）**

| 策略 | 原理 | 效果 |
|------|------|------|
| **Marlin Kernel** | 专门为 batch=1 优化的 INT4 GEMV | 3-4× 加速 |
| **AWQ/GPTQ** | Weight-only 量化，激活保持 FP16 | 2-3× 加速 |
| **FlashAttention** | 减少 Attention 的 HBM 访问 | 2-4× 加速 |
| **KV Cache** | 缓存历史 K/V | 必不可少 |
| **Speculative Decoding** | 小模型猜词，大模型验证 | 2-3× 加速 |
| **Continuous Batching** | 多请求 batch 合并 | 提高吞吐 |

### 标准参考答案（供参考）

**1. Key Cache 访问问题**：
- `t * kv_dim` 导致 strided access，相邻线程访问的内存地址间隔 `kv_dim`
- 当 seq_len 很长时（4K），cache miss 严重

**2. Softmax 效率问题**：
- pos=4000，256 线程，每个线程处理 4000/256 ≈ 16 个元素
- 但 exp、reduce 都是串行循环，没有充分利用并行性

**3. Value 加权问题（最慢部分）**：
- **每个线程串行遍历所有 position**：`for (int t = 0; t <= pos; t++)`
- 这是 O(seq_len) 的串行循环，完全无法并行
- 当 seq_len=4000 时，每个线程要做 4000 次内存访问

**4. FlashAttention 核心思想**：
- **Tiling**：将 Q、K、V 分块加载到 SRAM
- **Online Softmax**：增量计算 softmax，避免存储完整的 attention matrix
- **Memory**：从 O(N²) 降到 O(N)，避免 HBM 读写巨大的 attention matrix
